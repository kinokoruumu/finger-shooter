# テスト

## 基本方針

### テストの分類

| 分類 | 対象 | 例 |
|---|---|---|
| Unit test | 関数、小さい feature のロジック、単一コンポーネント | utils, hooks, 純粋なロジック関数 |
| Integration test | feature のまとまり、ページ単位のコンポーネント | 複数コンポーネントの連携、ページ全体の描画 |
| E2E test | 一連のユーザーフロー、バックエンドを含む統合テスト | ブラウザ上での操作フロー |

### テストファイルの配置

co-location 方式に従い、テスト対象の近くに配置する。

```
component-a/
├─ index.tsx
├─ hooks.ts
├─ utils.ts
├─ index.test.tsx       # コンポーネントのテスト
├─ hooks.test.ts        # フックのテスト
└─ utils.test.ts        # ユーティリティのテスト
```

### テストの書き方

- テストは実装の詳細ではなく振る舞いを検証する
- テストケース名は日本語で書く

### テストツールの選定

- テストツールはプロジェクトの技術スタックに合わせて選定する
- 不要なライブラリを先行インストールしない。実際にテストを書くタイミングで必要なものだけ導入する
- DOM環境（jsdom, happy-dom 等）はコンポーネントテストを書く段階で検討する

## テスト必須の対象

以下のコードはテストを書いてからUIに進む。テストなしでUI動作確認に頼らない。

| 対象 | 例 |
|---|---|
| utils（純粋関数） | 変換関数、計算ロジック、フォーマッタ |
| hooks のロジック部分 | 状態遷移、副作用の発火条件 |
| 状態に依存するビジネスロジック | グループ順次実行、スポーン計算 |

エッジケースとして以下を必ず検証する:
- 空入力（配列0個、null/undefined）
- 境界値（1グループ vs 複数グループ、ステップ0個）
- 旧データとの互換性（オプショナルフィールドが欠落した場合）

## プロジェクト固有

- Unit test のテストランナーは Vitest を使用（Vite ベースのプロジェクトのため）
- Integration test / E2E test のツールは未選定。必要になった段階で決める
